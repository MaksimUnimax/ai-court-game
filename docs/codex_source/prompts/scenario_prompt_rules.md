# Scenario Prompt Rules — AI Court Game

STATUS: draft_1_appendable

## Назначение

Этот документ собирает правила и условия для создания сценариев судебных дел.

Позднее эти правила станут основой GPT/API prompt.

Пока ChatGPT вручную создаёт сценарные документы, а пользователь тестирует их в draft MVP.

Документ должен пополняться по мере того, как выявляются проблемы сценариев.

## Основное правило

Сценарий должен быть решаемой головоломкой.

Недостаточно, чтобы дело было просто драматичным, смешным или странным.

Игрок должен иметь возможность прийти к правильному вердикту, кликая по участникам, вопросам и доказательствам.

## Обязательные качества сценария

У каждого сценария должно быть:

- чёткая скрытая истина;
- правильный вердикт;
- достаточно доказательств, чтобы поддержать правильный вердикт;
- хотя бы одно значимое противоречие или подсказка;
- отсутствие противоречий, делающих дело логически невозможным;
- отсутствие решения, которое требует информации, недоступной игроку;
- отсутствие зависимости от свободного текста;
- отсутствие необходимости в реальных юридических знаниях;
- отсутствие скрытого авторского reasoning.

## Ограничения draft MVP

Для первого draft MVP:

- без изображений;
- без OpenAI API;
- без сгенерированных портретов;
- без сгенерированных иллюстраций доказательств;
- без авторизации пользователя;
- без базы данных;
- без свидетелей по умолчанию;
- без свободного текстового ввода;
- весь геймплей через клики.

## Правило структуры сценария

Сценарий должен быть структурирован так, чтобы скрипт мог раскладывать контент по UI-блокам.

Сценарий не должен быть просто текстовой историей.

Он должен включать явные секции для:

- metadata;
- participants;
- evidence;
- questions;
- answers;
- unlock rules;
- verdict options;
- correct verdict;
- final explanation.

## Правила типов дел

Поддерживаемые типы дел для draft MVP:

1. Criminal case.
2. Dispute between two people.

Уголовное дело должно использовать варианты вердикта:

- guilty;
- not guilty.

Спор между двумя людьми должен использовать варианты вердикта:

- side A is right;
- side B is right.

## Правила участников

Для уголовного дела участники по умолчанию:

- accused person;
- prosecutor;
- defense lawyer.

Для спора между двумя людьми участники по умолчанию:

- side A;
- side B.

Свидетели в первом draft MVP не нужны, если они не добавлены позже отдельно.

## Правила вопросов

Вопросы должны быть заранее определены.

Игрок выбирает вопросы кликом по кнопкам.

Каждый вопрос принадлежит участнику.

Каждый ответ должен быть подготовлен заранее.

Ответы могут:

- раскрывать противоречие;
- подтверждать алиби;
- ослаблять алиби;
- разблокировать доказательство;
- разблокировать другой вопрос;
- честно вводить игрока в заблуждение.

Ответы не должны требовать ввода текста от игрока.

## Правила доказательств

В первом draft MVP доказательства должны быть текстовыми.

Каждое доказательство должно включать:

- id;
- title;
- visible description;
- detailed inspection text;
- whether it is available at start;
- unlock condition if hidden;
- what it proves;
- whether it is key evidence.

Доказательства должны иметь значение для головоломки.

Декоративных доказательств следует избегать.

## Правила разблокировки

Правила разблокировки должны быть детерминированными.

Хорошие условия разблокировки:

- после заданного конкретного вопроса;
- после открытия конкретного доказательства;
- после двух конкретных вопросов;
- после открытия доказательства и затем связанного вопроса.

Плохие условия разблокировки:

- случайность;
- скрытый таймер;
- author-only trigger;
- свободная текстовая фраза;
- неясное действие игрока.

## Правила честности

Сценарий нечестный, если:

- правильный ответ зависит от информации, которой не было показано;
- два вердикта одинаково обоснованы;
- финальное объяснение добавляет новые факты, которых не было;
- доказательства противоречат скрытой истине;
- игрок должен угадывать намерение автора;
- ответ участника ломает установленную временную линию.

## Правила финального объяснения

После вердикта сценарий должен объяснять:

- правильный вердикт;
- что произошло на самом деле;
- ключевое доказательство;
- ключевое противоречие;
- почему выбранный неправильный вердикт неверен;
- что игрок должен был заметить.

## Будущее API-направление

Когда позже будет добавлена GPT/API generation, prompt должен требовать структурированный вывод.

Future GPT response should be strict JSON.

Future GPT response should include:

- scenario metadata;
- anti-repeat tags;
- hidden truth;
- participants;
- evidence;
- questions;
- unlock rules;
- verdict options;
- correct verdict;
- solution explanation;
- validation checklist.

Будущий GPT prompt должен явно запрещать неразрешимые случаи.

## Вопросы для уточнения позже

Эти правила будут уточняться после playtesting:

- best scenario JSON format;
- number of evidence items per difficulty;
- number of questions per participant;
- how to mark key contradictions;
- how to validate solvability;
- how to prevent repeated plots;
- how to convert manual scenario documents into GPT API prompts.

## Addendum — Scenario graph rules

Сценарий должен генерироваться как data для универсального scenario graph engine.

Сценарий не должен предполагать фиксированный порядок диалогов, hardcoded в app.

Сценарий должен явно определять:

- detailed participants;
- relationships between participants;
- dialogue actions;
- evidence actions;
- availability conditions;
- effects after actions;
- unlock dependencies;
- discovered facts;
- contradictions;
- verdict options;
- correct verdict;
- final explanation.

Каждый участник должен быть больше, чем просто label роли.

Данные участника должны включать:

- name;
- role;
- job or position;
- relation to the case;
- relation to other participants;
- public motive or conflict;
- what the participant knows;
- what the participant hides or misrepresents if relevant.

Каждое dialogue action должно определять, когда оно доступно и что оно меняет.

Каждое evidence item должно определять, когда оно доступно и что оно доказывает.

Сценарий не должен требовать нового code для конкретного случая.

Сценарий должен исполняться тем же engine, что и любой другой кейс.

Будущий GPT prompt должен требовать от модели выводить scenario graph data, а не обычный рассказ.

## Addendum — Реальные факты как скрытый инструмент сценарной загадки

Сценарии AI Court Game могут использовать реальные факты нашего мира как часть судебной загадки.

Это не отдельная кнопка интерфейса и не отдельная видимая механика.

Это правило для создания сценариев и будущего prompt для LLM/API.

Игрок-судья находится в браузере и может сам вручную проверить факт в интернете, если посчитает это нужным.

Игра не должна подключать Google, web search или внешний API в черновом MVP.

### Правило мира

Игровой мир пересекается с реальным миром.

Любой факт, который действительно был или есть в нашем мире, считается истинным и в мире игры.

При этом игровой мир может быть шире нашего мира и дополнительно включать:

- вымышленные события;
- комичные элементы;
- магию;
- sci-fi;
- абсурдные ситуации;
- полностью вымышленных персонажей и конфликты.

Но если сценарий ссылается на реальный проверяемый факт нашего мира, этот факт должен работать в игре так же, как в реальности.

### Как LLM должна использовать это правило

LLM может строить часть загадки на проверяемом утверждении участника.

Примеры проверяемых утверждений:

- участник говорит, что был на концерте реальной группы в конкретную дату;
- участник ссылается на расписание публичного события;
- участник утверждает, что был в городе, куда в этот день не ходил нужный рейс;
- участник ссылается на публично известный закон, дату, праздник, матч, выставку, премьеру, историческое событие;
- участник называет факт из биографии публичного человека или организации;
- участник ошибается в географии, календаре, времени, расписании или публичной хронологии.

Игрок может самостоятельно проверить такой факт в интернете и понять, говорит ли участник правду.

### Важно

Сценарий не должен прямо говорить игроку: "это механика внешней проверки".

Сценарий должен естественно встроить проверяемый факт в:

- алиби;
- показания;
- спор;
- объяснение мотива;
- дату;
- маршрут;
- расписание;
- публичное событие;
- документ;
- биографическую или историческую ссылку.

Игрок сам решает, стоит ли проверять это утверждение.

### Требования честности

Если сценарий использует реальный факт как ключ к решению, этот факт должен быть:

- достаточно стабильным;
- публично проверяемым;
- не слишком спорным;
- не зависящим от закрытого источника;
- не требующим платного доступа;
- не требующим экспертных знаний;
- сформулированным достаточно конкретно: дата, место, событие, организация, человек или расписание.

Нельзя строить ключевую разгадку на факте, который:

- невозможно быстро проверить;
- имеет противоречивые источники;
- мог измениться после создания сценария;
- требует глубокого профессионального исследования;
- не имеет понятной связи с делом.

### Скрытая структура для будущего LLM prompt

В будущем prompt для LLM должен требовать, чтобы сценарий внутренне фиксировал:

- какое утверждение участника является проверяемым;
- какой реальный факт должен проверить игрок;
- какой ожидаемый результат проверки;
- как этот результат влияет на вердикт;
- где это объясняется в финальном разборе.

Эта информация может быть скрыта от игрока до финального объяснения.

Игрок должен видеть только естественную реплику, улику или обстоятельство дела.

### Пример

Плохо:

"Нажмите кнопку внешней проверки, чтобы узнать, что подсудимый соврал."

Хорошо:

Подсудимый говорит:

"12 января я был на концерте группы 'Руки Вверх' в этом городе, поэтому никак не мог быть на месте преступления."

Сценарий должен позволить игроку заметить, что это алиби можно проверить самостоятельно.

Если игрок проверит расписание и увидит, что такого концерта не было, он поймёт, что участник лжёт.

Финальный разбор должен объяснить:

- какое утверждение было проверяемым;
- что именно было неверно;
- почему это ломает алиби;
- как это связано с правильным вердиктом.

## Addendum — Дела должны быть необычными и запоминающимися

Сценарии AI Court Game не должны быть скучными повседневными спорами.

Даже если основа дела бытовая или гражданская, в деле должен быть хотя бы один элемент "из ряда вон".

Этот элемент может быть:

- абсурдным;
- странным;
- гротескным;
- комичным;
- тревожным;
- магическим;
- sci-fi;
- сюрреалистичным;
- социально нелепым;
- юридически необычным;
- бытовым, но доведённым до странного конфликта.

Необычный элемент не обязан быть юмористическим.

Необычный элемент не обязан определять правильный вердикт.

Его задача:

- сделать дело запоминающимся;
- добавить вкус и разнообразие;
- не дать сценарию превратиться в скучное сравнение документов;
- усилить желание игрока дочитать и разобраться.

### Разделение необычности и загадки

Необычный элемент и детективная загадка — не одно и то же.

Сценарий должен иметь:

1. Необычный элемент для повествовательного интереса.
2. Честную загадку для решения дела.

Необычный элемент может не влиять на итоговый вердикт.

Детективная загадка должна влиять на вывод игрока.

### Пример необычного бытового дела

Скучно:

"Арендатор и арендодатель спорят о залоге. Есть договор, акт и чек."

Лучше:

"Арендаторша всё оплатила, мебель целая, но арендодатель удерживает часть залога, потому что она громко пердела по ночам. Соседи оставили отзыв на сайте аренды, написали, что в квартире живут пердуны и по ночам невозможно спать. Арендодатель утверждает, что из-за этого будущие клиенты будут реже бронировать жильё, значит он понёс убыток."

Правильное решение:

Залог нужно вернуть, потому что в договоре не было условия о таком поведении, имущество не повреждено, а репутационный риск арендодателя не является основанием удерживать залог.

В этом примере странный элемент делает дело живым, но не обязан быть детективным ключом.

Детективный ключ должен быть добавлен отдельно: противоречие, скрытая улика, проверяемое утверждение, ложная трактовка или другой честный механизм решения.

### Требование к LLM

LLM должна избегать полностью обыденных дел.

Если дело кажется слишком стандартным, LLM должна добавить необычный, странный или запоминающийся элемент, не ломая честность вердикта.

При этом LLM не должна превращать каждое дело в случайный хаос.

Необычность должна быть связана с конфликтом, участниками, мотивацией, последствиями или атмосферой дела.

## Addendum — Иллюстрации как скрытая дедуктивная часть сценария

Иллюстрации в AI Court Game должны быть частью решения дела, но не должны прямо подсказывать игроку, куда смотреть.

Нельзя строить основную механику на явных подсветках, стрелках, красных кругах, подписанных уликах или очевидных hotspot-зонах.

Иллюстрация должна выглядеть как естественная сцена, портрет, место преступления, предмет спора или важный объект дела.

Важные детали должны быть вписаны в изображение естественно.

Игрок-судья сам должен вглядываться, замечать странности и сопоставлять их с показаниями, обстоятельствами дела и логикой.

### Решение через связку источников

Правильный вывод не должен находиться только из диалогов.

Правильный вывод не должен находиться только из иллюстраций.

Сценарий должен быть устроен так, чтобы честное решение требовало сочетания:

- диалогов;
- визуальных деталей на иллюстрациях;
- логики дела;
- иногда внешне проверяемых реальных фактов.

Если игрок читает только диалоги, ему должно не хватать важной визуальной детали.

Если игрок смотрит только иллюстрации, ему должно не хватать контекста из показаний.

Если игрок пытается угадать без сопоставления, он может выбрать случайно, но сценарий не должен поощрять такое прохождение.

### Дедуктивный принцип

Иллюстрация должна поддерживать ход:

1. Наблюдение.
2. Сопоставление.
3. Вывод.

Пример:

- игрок замечает деталь на изображении;
- вспоминает, что участник говорил в диалоге;
- видит несоответствие;
- делает вывод, что версия участника ложная или неполная.

Это близко к принципу наблюдения и дедукции: маленькая физическая деталь получает значение только после сопоставления с другими фактами.

### Минимум иллюстраций

Каждое полноценное дело должно предусматривать минимум три новые иллюстрации:

- портрет или визуальный образ первого важного участника;
- портрет или визуальный образ второго важного участника;
- место преступления, место спора, ключевая сцена или ключевой предмет.

Иллюстраций может быть больше, если дело требует:

- отдельный предмет;
- второй ракурс сцены;
- документ;
- место события;
- визуальную деталь, которую нельзя честно показать в общей сцене.

### Иллюстрации не являются украшением

Иллюстрация не должна быть просто декоративной.

Каждая ключевая иллюстрация должна содержать хотя бы одну деталь, связанную с делом.

Деталь может:

- подтверждать часть показаний;
- противоречить словам участника;
- указывать на неправильную трактовку события;
- связывать участника с местом;
- показывать физическую невозможность версии;
- намекать на скрытый порядок событий;
- давать контекст для финального вывода.

### Двухэтапная генерация

В production-версии генерация должна идти двумя этапами.

Первый этап: генерация сценария.

Сценарный prompt должен создать:

- дело;
- скрытую правду;
- участников;
- диалоги;
- зависимости;
- правильный вердикт;
- финальный разбор;
- план иллюстраций;
- список важных визуальных деталей;
- объяснение, как визуальные детали связаны с решением.

Второй этап: генерация иллюстраций.

На основе готового сценария второй prompt должен создать промпты для изображений.

Промпты изображений должны указывать:

- что должно быть изображено;
- стиль;
- композицию;
- какие детали важны для дела;
- какие детали должны быть заметны, но не выделены искусственно;
- какие детали нельзя потерять при генерации.

Иллюстрационный prompt не должен просить модель подписывать улики или выделять их стрелками.

### Текущий MVP

В текущем MVP API-генерации ещё нет.

Пока ChatGPT по запросу пользователя вручную готовит сценарий и отдельные prompts/описания для иллюстраций.

Пользователь загружает подготовленные материалы в игру.

Скрипт должен в будущем распределять иллюстрации по местам сценария автоматически по данным сценария.

### visual_assets в сценарии

Сценарный prompt может дополнительно описывать раздел visual_assets.

Этот раздел должен связывать изображения с участниками, сценами, объектами или делом в целом.

Для каждого визуального элемента prompt должен задавать:

- id;
- type;
- file;
- target_type или placement;
- title;
- alt;
- скрытое назначение визуальной детали.

Скрытое назначение не должно быть показано игроку в интерфейсе.

Файловые имена должны быть пригодны для сопоставления с загруженными изображениями в браузере.

Это нужно, чтобы позже один загруженный пакет дела можно было разложить по нужным местам автоматически.

## Schema compliance gate before delivery

Before giving a scenario file to the user or packaging a case, validate schema compatibility against the current loader. Do not improvise field names, effect shapes, or visual asset types. If the loader expects an exact field name or exact type name, use that exact name.

### Required top-level sections

Required top-level sections are:

- `metadata`
- `case_intro`
- `participants`
- `relationships`
- `evidence`
- `dialogue_actions`
- `verdicts`
- `solution`

`visual_assets` is allowed and recommended when the case includes cover art, portraits, scenes, or object illustrations.

### visual_assets rules

Use only supported visual asset types:

- `cover`
- `participant_portrait`
- `scene`
- `object`

Do not use unsupported type names such as:

- `case_cover`
- `case_illustration`

Each `visual_assets` item must define at least one of:

- `target_type`
- `placement`

Best practice: define both when possible.

Recommended mapping:

- case cover: `type: "cover"`, `target_type: "case"`, `placement: "cover"`
- participant portrait: `type: "participant_portrait"`, `target_type: "participant"`, `placement: "participant_card"`
- case scene image: `type: "scene"`, `placement: "case_panel"`
- object/evidence-style image: `type: "object"`, `placement: "case_panel"`

### participants rules

Each participant must include at minimum:

- `id`
- `name`
- `role`
- `position`
- `relation_to_case`
- `public_description`
- `relationships`

Optional richer fields such as `voice_direction`, `hidden_truth`, or other authoring fields may be added, but they never replace the required fields.

### relationships rules

Each relationship must include:

- `id`
- `from_participant_id`
- `to_participant_id`
- `label`
- `description`

`from_participant_id` and `to_participant_id` must reference existing participant ids.

Do not replace these with aliases such as `from` or `to`.

### evidence rules

Use `inspection_text` as the detailed readable evidence body.

Do not replace it with:

- `detail`
- `details`
- any other alias

Recommended minimum evidence fields:

- `id`
- `title`
- `short_description`
- `inspection_text`
- `proves`

If `effects` are used, `effects` must be an array of objects, not strings.

Correct pattern example:

JSON array:
[
  {
    "type": "show_note",
    "note": "..."
  }
]

Incorrect pattern example:

JSON array:
[
  "some text note"
]

### dialogue_actions rules

Use `response_text` as the participant answer field.

Do not replace it with:

- `response`
- `answer`
- any other alias

Each `dialogue_actions` item should include at minimum:

- `id`
- `participant_id`
- `label`
- `response_text`

`participant_id` must reference an existing participant.

If `effects` are used, `effects` must be an array of objects, not strings.

### solution rules

`solution` must include:

- `explanation`

Recommended additional fields:

- `correct_verdict_id`
- `truth`
- `key_points`
- `wrong_verdict_explanation`

Do not ship a scenario where `solution.explanation` is missing.

### Pre-delivery loader-compatibility checklist

Before delivering any scenario JSON, verify all of the following:

- all required participant fields are present
- all `from_participant_id` and `to_participant_id` values point to existing participants
- all evidence entries use `inspection_text`
- all dialogue actions use `response_text`
- all `effects` entries are objects, not strings
- all `visual_assets` entries use only supported types
- every visual asset defines `target_type` or `placement`
- `solution.explanation` is present
- the final JSON parses successfully
- the final JSON is locally validated before delivery

### Non-negotiable rule

When creating scenario files for this project, follow the loader schema exactly. Do not rename fields for style, convenience, or readability. If the loader expects `response_text`, `inspection_text`, `cover`, `participant_portrait`, `scene`, or `object`, use those exact names.
